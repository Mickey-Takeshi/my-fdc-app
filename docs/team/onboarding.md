### 2.2 ドキュメントオーナー

| ドキュメント領域 | オーナー | 責務 |
|----------------|---------|------|
| FDC-CORE.md | テックリード | 常に最新の状態を維持 |
| guides/ | 担当開発者 | 新しい知見を随時追記 |
| runbooks/ | 実装担当者 | Phase 完了時に作成 |
| adr/ | 提案者 | 決定内容の記録・更新 |
| team/ | テックリード | プロセス変更時に更新 |
| strategy/ | プロダクトマネージャー | 戦略変更時に更新 |
| troubleshooting/ | 遭遇した開発者 | 問題解決時に追記 |

### 2.3 ドキュメント更新ルール

| ルール | 内容 |
|-------|------|
| 更新タイミング | 機能追加・変更時に必ず更新 |
| レビュー | ドキュメント変更も PR レビュー対象 |
| 鮮度チェック | 四半期ごとに全ドキュメントの棚卸し |
| 形式 | Markdown 形式。テンプレートに従う |
| 言語 | 日本語（コード・コマンドは英語） |
| 廃止処理 | 不要になったドキュメントは `[DEPRECATED]` をタイトルに付与 |

### 3.3 ローテーションスケジュール

| 項目 | 内容 |
|------|------|
| ローテーション頻度 | 四半期ごと（1月・4月・7月・10月） |
| 目的 | 特定メンバーへの依存を防ぎ、知識を分散させる |
| 手順 | CODEOWNERS ファイルを更新し、PR で変更をレビュー |
| 引き継ぎ | ローテーション時にペアプログラミングセッションを実施 |
| 記録 | ローテーション履歴を `docs/team/` に記録 |

## 4. ペアプログラミング / モブプログラミング

### 4.1 適用シナリオ

| シナリオ | 推奨度 | 理由 |
|---------|:------:|------|
| 新メンバーのオンボーディング | 高 | コードベースの理解を加速。質問しやすい環境 |
| 複雑な機能の設計・実装 | 高 | 複数の視点で品質向上。設計ミスの早期発見 |
| バグ調査 | 高 | 2人で調査することで原因特定を高速化 |
| 単純な作業（ドキュメント更新等） | 低 | 個人作業の方が効率的 |
| 定型的なCRUD実装 | 低 | テンプレートに従えば個人で十分 |

### 4.2 実施ルール

| ルール | 内容 |
|-------|------|
| 頻度 | 週 1-2 回 |
| 時間 | 1 セッション 1-2 時間 |
| ロール交代 | 15-30 分ごとにドライバー/ナビゲーターを交代 |
| 休憩 | 1 時間ごとに 10 分休憩 |
| 事前準備 | ゴール・対象コードを事前に共有 |
| 振り返り | セッション終了時に 5 分間の振り返り |

### 4.3 ロール定義

| ロール | 担当 | 行動指針 |
|-------|------|---------|
| **ドライバー** | キーボード操作を担当 | コードを書くことに集中する。ナビゲーターの指示に従う |
| **ナビゲーター** | 方向性・設計の指示を担当 | 全体像を考え、次のステップを提案する。コードの品質を監視する |

### 4.4 モブプログラミング（3人以上）

- 複雑な設計判断やアーキテクチャ変更時に実施
- 1人がドライバー、残りがナビゲーター
- ドライバーのローテーションは 15 分間隔
- 全員がコードに対する共通理解を持つことが目的

## 5. メンター制度

### 5.1 期間別メンタリング

| 期間 | メンタリング形式 | 頻度 | 内容 |
|------|----------------|------|------|
| **1週目** | デイリー 1on1 | 毎日 15-30分 | 環境セットアップ支援、質問対応、コードベース案内 |
| **Month 1** | ウィークリー 1on1 | 週1回 30分 | 進捗確認、技術的な疑問の解消、PR レビューの解説 |
| **Month 2-3** | バイウィークリー 1on1 | 隔週 30分 | キャリア相談、技術的な深掘り、自律的な開発への移行支援 |
| **Month 4 以降** | オンデマンド | 必要に応じて | 困ったときの相談相手として継続 |

### 5.2 メンター選定基準

| 基準 | 内容 |
|------|------|
| 技術力 | FDC のコードベースに精通していること |
| 経験 | プロジェクト参加から 3 か月以上の経験 |
| コミュニケーション | 技術的な内容を分かりやすく説明できること |
| 余力 | メンタリングに時間を割ける業務状況であること |
| 多様性 | メンティーと異なる専門領域を持つことが望ましい |

### 5.3 メンタリングで扱うトピック

**技術面**:
- FDC のアーキテクチャと設計思想
- TypeScript / React / Next.js のベストプラクティス
- Supabase の使い方と注意点
- テスト戦略とデバッグ手法
- パフォーマンス・セキュリティの考慮事項

**プロセス面**:
- Git / GitHub のワークフロー
- コードレビューの受け方・出し方
- ドキュメントの書き方
- Issue / PR の書き方

**チーム面**:
- チームの文化・価値観
- コミュニケーションの取り方
- 困ったときの相談先
