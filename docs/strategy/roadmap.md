## 2. リリース戦略

### 2.1 リリースフロー

```
Internal Test → Closed Beta → Open Beta → GA（General Availability）
    |               |              |            |
    v               v              v            v
  社内テスト     招待制テスト    公開テスト     正式リリース
  (1-2週間)     (2-4週間)      (2-4週間)      (継続)
```

### 2.2 各フェーズの詳細

#### Internal Test（社内テスト）

| 項目 | 内容 |
|------|------|
| **期間** | 1-2週間 |
| **対象** | 開発チーム + 社内メンバー |
| **人数** | 5-10名 |
| **目的** | 重大バグの検出、UXの確認 |
| **フィードバック方法** | Slack チャンネル / Issue |
| **終了基準** | P0/P1 バグ 0件 |

#### Closed Beta（クローズドベータ）

| 項目 | 内容 |
|------|------|
| **期間** | 2-4週間 |
| **対象** | 招待制の外部ユーザー |
| **人数** | 20-50名 |
| **目的** | 実環境でのUX検証、フィードバック収集 |
| **フィードバック方法** | アンケート / インタビュー / アプリ内フィードバック |
| **終了基準** | NPS 20+, P0/P1 バグ 0件, 満足度 70%+ |

#### Open Beta（オープンベータ）

| 項目 | 内容 |
|------|------|
| **期間** | 2-4週間 |
| **対象** | 誰でも参加可能 |
| **人数** | 100-500名 |
| **目的** | スケーラビリティ検証、最終調整 |
| **フィードバック方法** | アプリ内フィードバック / SNS モニタリング |
| **終了基準** | エラー率 0.1%未満, レスポンスタイム p95 < 500ms |

#### GA（General Availability）

| 項目 | 内容 |
|------|------|
| **対象** | 全ユーザー |
| **サポート** | フルサポート（SLA 適用） |
| **課金** | 本番課金開始 |
| **SLA** | 稼働率 99.9% |

### 2.3 リリース基準テーブル

| 基準 | Internal | Closed Beta | Open Beta | GA |
|------|----------|-------------|-----------|-----|
| P0 バグ | 0件 | 0件 | 0件 | 0件 |
| P1 バグ | 0件 | 0件 | 0件 | 0件 |
| P2 バグ | 許容 | 3件以下 | 1件以下 | 0件 |
| テストカバレッジ | 60%+ | 70%+ | 80%+ | 80%+ |
| パフォーマンス（p95） | < 1s | < 800ms | < 500ms | < 500ms |
| エラー率 | < 1% | < 0.5% | < 0.1% | < 0.1% |
| ドキュメント | Draft | Draft | Review済み | 公開済み |
| セキュリティレビュー | - | 実施済み | 実施済み | 実施済み |
| 負荷テスト | - | - | 実施済み | 実施済み |

## 3. Feature Flag 運用ガイドライン

### 3.1 Feature Flag の種類

| 種類 | 用途 | 寿命 | 例 |
|------|------|------|-----|
| **Release Flag** | 新機能のリリース制御 | 短期（数週間） | `feature_new_dashboard` |
| **Experiment Flag** | A/Bテスト | 中期（数ヶ月） | `experiment_pricing_v2` |
| **Ops Flag** | 運用時の緊急スイッチ | 長期（常設） | `kill_switch_sync` |
| **Permission Flag** | プラン別機能制御 | 永続 | `plan_pro_analytics` |

### 3.2 命名規則

```
{type}_{feature_name}

例:
- release_action_map_v2
- experiment_onboarding_flow
- ops_disable_google_sync
- plan_enterprise_audit_log
```

### 3.3 ライフサイクル

```
作成 → テスト → 段階的ロールアウト → 全展開 → クリーンアップ
                  |
                  ├── 10% → 25% → 50% → 100%
                  |
                  └── 問題発生時 → 即座にOFF → 調査 → 修正
```

### 3.4 クリーンアップルール

- Release Flag: GA後 2週間以内に削除
- Experiment Flag: 結論が出たら 1週間以内に削除
- 未使用フラグの定期棚卸し: 月次

## 4. 技術負債管理

### 4.1 20% ルール

各スプリントの**20%のキャパシティ**を技術負債解消に充てる。

```
スプリントキャパシティ: 10人日の場合
├── 新機能開発: 8人日（80%）
└── 技術負債解消: 2人日（20%）
```

### 4.2 技術負債リスト

| # | 負債内容 | 影響度 | 緊急度 | 見積もり | 状態 | 対応予定 |
|---|---------|--------|--------|---------|------|---------|
| 1 | | 高/中/低 | 高/中/低 | ___日 | 未着手/進行中/完了 | Q_ |
| 2 | | 高/中/低 | 高/中/低 | ___日 | | |
| 3 | | 高/中/低 | 高/中/低 | ___日 | | |
| 4 | | 高/中/低 | 高/中/低 | ___日 | | |
| 5 | | 高/中/低 | 高/中/低 | ___日 | | |

### 4.3 技術負債の分類

| カテゴリ | 説明 | 例 |
|---------|------|-----|
| **コード品質** | 読みにくい・複雑なコード | 長大関数、重複コード |
| **アーキテクチャ** | 設計上の問題 | 密結合、スケーラビリティ |
| **依存関係** | ライブラリの更新 | EOLパッケージ、脆弱性 |
| **テスト** | テスト不足 | カバレッジ低下、E2Eなし |
| **インフラ** | インフラの問題 | 手動デプロイ、監視不足 |
| **ドキュメント** | ドキュメント不足 | API仕様、アーキテクチャ図 |

### 4.4 レビューケイデンス

| レビュー | 頻度 | 内容 |
|---------|------|------|
| 負債トリアージ | 隔週 | 新規負債の分類・優先度付け |
| 負債スプリント | 月次 | 1スプリント丸ごと負債解消に充てる |
| 負債監査 | 四半期 | 負債リストの棚卸し・戦略レビュー |
