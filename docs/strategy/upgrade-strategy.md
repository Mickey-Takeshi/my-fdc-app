> Free ユーザーを Pro へ、月払いを年払いへ、Pro を Enterprise へ自然にアップグレードさせるための戦略テンプレート。

## 2. 機能制限テーブル（自然なアップグレードナッジ）

ユーザーが制限に到達した際に、自然にアップグレードを検討するポイントを設計する。

### 2.1 制限設計

| 機能 | Free 制限 | 制限到達の目安 | ナッジタイミング | Pro で解除 |
|------|---------|-------------|----------------|-----------|
| ワークスペース | 1個 | 即座 | 2個目作成時 | 5個まで |
| メンバー | 3名 | 1-2週間 | 4人目招待時 | 20名まで |
| リード管理 | 50件 | 2-4週間 | 40件到達時 | 無制限 |
| ストレージ | 100MB | 1-2ヶ月 | 80MB到達時 | 10GB |
| API 呼び出し | 100/日 | 利用者による | 80/日到達時 | 10,000/日 |
| OKR管理 | 利用不可 | - | 機能発見時 | 全機能 |
| Google連携 | 利用不可 | - | 設定画面アクセス時 | 全機能 |
| 監査ログ | 利用不可 | - | 管理画面アクセス時 | 30日間 |
| データエクスポート | CSV のみ | 必要時 | エクスポート時 | CSV + JSON + PDF |

### 2.2 制限設計の原則

```
1. 基本的な価値は Free で体験できる
2. ユーザーが成長（データ増加・チーム拡大）すると自然に制限に到達
3. 制限到達時に明確な価値提案を表示
4. 制限はイライラさせるのではなく、次のステップを示す
5. 制限に到達するまでの期間は 2-4週間 が理想
```

| インセンティブ | 内容 | 効果 |
|-------------|------|------|
| **2ヶ月分無料** | 年払い = 10ヶ月分の価格 | ストレートな割引 |

| タイミング | アプローチ | 期待効果 |
|-----------|----------|---------|
| **サインアップ時** | 月払い/年払いトグルを表示（年払いにデフォルト設定しない） | 初期選択で年払い獲得 |

| **月払い更新時** | 更新確認画面で年払いの節約額を表示 | 更新タイミングでの切り替え |
| **機能追加時** | 「年払いユーザーは新機能を先行利用」告知 | FOMO効果 |
| **年末/年度末** | 期間限定の年払い特別割引 | 季節的インセンティブ |
