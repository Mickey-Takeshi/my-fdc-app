# 機能優先順位付けテンプレート

> RICE フレームワークを用いて機能の優先順位を客観的に決定するためのテンプレート。
> プロダクトバックログの整理と意思決定の透明性を高める。

---

## 1. RICE フレームワーク

### 1.1 RICEとは

RICE は Intercom が考案した優先順位付けフレームワーク。
主観に頼らず、定量的にフィーチャーの優先度を決定できる。

| 要素 | 意味 | 説明 | スケール |
|------|------|------|---------|
| **R**each | リーチ | 一定期間内に影響するユーザー数 | 人数/四半期 |
| **I**mpact | インパクト | 各ユーザーへの影響度 | 3=大 / 2=高 / 1=中 / 0.5=低 / 0.25=極小 |
| **C**onfidence | 確信度 | 見積もりの確度 | 100%=高 / 80%=中 / 50%=低 |
| **E**ffort | 工数 | 実装に必要な人月 | 人月（小数可） |

### 1.2 RICE スコア計算式

```
RICE Score = (Reach x Impact x Confidence) / Effort
```

### 1.3 スコアリング例

```
機能: ダッシュボード改善
- Reach: 500 ユーザー/四半期
- Impact: 2（高）
- Confidence: 80%
- Effort: 2 人月

RICE Score = (500 x 2 x 0.8) / 2 = 400
```

## 3. 優先度マトリクス

### Impact vs Effort マトリクス

```
高 Impact
    |
    |  Big Bet              Quick Win
    |  (大きな賭け)          (すぐやる)
    |
    |  高インパクト           高インパクト
    |  高コスト               低コスト
    |  → 計画的に実行         → 最優先で実行
    |
    |─────────────────────────────────────
    |
    |  Avoid                 Fill In
    |  (やらない)            (余裕があれば)
    |
    |  低インパクト           低インパクト
    |  高コスト               低コスト
    |  → 見送り               → バッファで対応
    |
    └──────────────────────────────────── 低 Effort → 高 Effort
```

### 各象限のアクション

| 象限 | 条件 | アクション | 例 |
|------|------|-----------|-----|
| **Quick Win** | 高Impact + 低Effort | 最優先で実行 | UIの微調整、バグ修正 |
| **Big Bet** | 高Impact + 高Effort | ロードマップに計画 | 新機能、大型リファクタ |
| **Fill In** | 低Impact + 低Effort | スプリントのバッファで | 小さな改善、文言修正 |
| **Avoid** | 低Impact + 高Effort | やらない判断 | 低需要な大型機能 |

## 5. 優先順位レビュープロセス

### レビュー頻度

| レビュー | 頻度 | 参加者 | 目的 |
|---------|------|--------|------|
| バックログリファイン | 週次 | PM + エンジニア | 新規アイテム追加・スコア更新 |
| 優先度レビュー | 隔週 | PM + デザイナー + エンジニアリード | 優先順位の見直し |
| ロードマップレビュー | 月次 | 経営 + PM + エンジニアリード | 四半期目標との整合性確認 |
| 戦略レビュー | 四半期 | 経営チーム全体 | 方向性・リソース配分の決定 |

### スコア更新トリガー

以下のイベント発生時にRICEスコアを再計算する:

- [ ] ユーザーインタビューの結果が得られた
- [ ] 定量データ（利用率、解約理由）が更新された
- [ ] 競合が類似機能をリリースした
- [ ] 技術的な前提条件が変わった（工数の見直し）
- [ ] ビジネス目標が変更された

## 6. RICE 以外の判断基準

RICE スコアだけでは判断できない場合の補助的な基準。

| 基準 | 説明 | 重み |
|------|------|------|
| **戦略的整合性** | 会社のビジョン・ミッションとの一致度 | 高 |
| **技術的負債解消** | 放置するとコストが増大するか | 中 |
| **法的/コンプライアンス** | 対応必須の規制要件か | 最高（必須） |
| **セキュリティ** | 脆弱性対応の緊急度 | 最高（必須） |
| **顧客約束** | 大口顧客への約束事項か | 高 |
| **チームモチベーション** | 開発チームの士気に影響するか | 低 |
