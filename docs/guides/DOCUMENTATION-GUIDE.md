# Documentation Guide

> FDC Modular Starter のドキュメント運用・管理ガイド

---

## 1. ドキュメント階層

FDC プロジェクトのドキュメントは以下の階層構造を持つ。
上位ドキュメントが下位ドキュメントの方針を決定し、矛盾が生じた場合は上位を正とする。

```
FDC-CORE.md              ... 最上位コンパス（開発方針・フェーズ管理・用語定義）
  |
  +-- DEVELOPMENT.md      ... 技術詳細（アーキテクチャ・コーディング規約・API 仕様）
  |
  +-- CHANGELOG.md         ... 変更履歴（Keep a Changelog 形式）
  |
  +-- guides/              ... 個別ガイド（セキュリティ・デプロイ・移行 etc.）
  |
  +-- runbooks/            ... 実装手順書（フェーズ単位のステップバイステップ）
  |
  +-- specs/               ... 仕様書（技術スタックバージョン etc.）
  |
  +-- sql/                 ... SQL リファレンス（RLS・メンテナンス etc.）
```

### 各ドキュメントの役割

| ドキュメント | 役割 | 読者 |
|-------------|------|------|
| `FDC-CORE.md` | プロジェクトの羅針盤。全セッションの起点 | 全員（人間 + AI） |
| `DEVELOPMENT.md` | 技術的な実装ルール・コーディング規約 | 開発者 |
| `CHANGELOG.md` | リリースごとの変更記録 | 全員 |
| `guides/*.md` | 特定トピックの詳細ガイド | 必要に応じて参照 |
| `runbooks/*.md` | フェーズ実装の手順書 | 実装担当者 |

### 参照の優先順位

1. **FDC-CORE.md** -- 最上位。開発方針・フェーズ状況・用語定義
2. **DEVELOPMENT.md** -- 技術詳細。コーディング規約・ディレクトリ規約
3. **個別ガイド** -- トピック別の深掘り（セキュリティ、パフォーマンス等）
4. **ランブック** -- フェーズ実装の具体的手順

矛盾が生じた場合: FDC-CORE.md > DEVELOPMENT.md > 個別ガイド の順で正とする。

## 2. ドキュメント更新タイミング

### 必須更新タイミング

| タイミング | 更新対象 | 内容 |
|-----------|---------|------|
| 機能追加時 | FDC-CORE.md + CHANGELOG.md | フェーズ状況の更新、変更内容の記録 |
| フェーズ完了時 | FDC-CORE.md + CHANGELOG.md | フェーズテーブルの状態更新、バージョン番号の更新 |
| バージョンアップグレード時 | FDC-CORE.md + CHANGELOG.md + 技術スタック表 | 依存関係のバージョン反映 |
| セキュリティ変更時 | SECURITY.md + CHANGELOG.md + FDC-CORE.md | 脆弱性対応・ポリシー変更の記録 |
| バグ修正時 | CHANGELOG.md | Fixed セクションに記録 |
| アーキテクチャ変更時 | DEVELOPMENT.md + FDC-CORE.md | 構造変更の反映 |

### 更新チェックリスト

作業完了時に以下を確認する:

- [ ] CHANGELOG.md に変更内容を追記したか
- [ ] FDC-CORE.md のフェーズテーブルを更新したか
- [ ] FDC-CORE.md のバージョン番号を更新したか
- [ ] 関連する個別ガイドを更新したか
- [ ] `npm run build` が成功するか

## 3. ランブックテンプレート

ランブックはフェーズ実装の再現可能な手順書である。以下のテンプレートに従って作成する。

### テンプレート

```markdown
# Phase N: タイトル

| 項目 | 内容 |
|------|------|
| 日付 | YYYY-MM-DD |
| 対象バージョン | vX.Y.Z |
| 見積もり時間 | XX 分 |
| ステータス | 未着手 / 進行中 / 完了 / 中止 |

---

## 目的（Purpose）

このフェーズで達成する目標を 1-3 文で記述する。

---

## 現状分析（Current Analysis）

実装前の現在の状態を記述する。
- 既存の関連コード・ファイル
- 現在の制約・問題点
- 依存関係

---

## 実装ステップ（Implementation Steps）

### Step 1: タイトル

説明文。

ファイル: `path/to/file.ts`

（コードブロック）

### Step 2: タイトル

説明文。

（コードブロック）

- [ ] すべての実装ステップが完了している

- [ ] CHANGELOG.md が更新されている
- [ ] FDC-CORE.md のフェーズ状況が更新されている
- [ ] コミットメッセージが規約に従っている

---

## 変更ログ（Change Log）

| 日付 | 変更内容 |
|------|---------|
| YYYY-MM-DD | 初版作成 |
```

### 良いランブックの特徴

1. **再現可能（Reproducible）**: 別の開発者（または AI エージェント）が同じ手順で同じ結果を得られる
2. **チェックリスト形式**: 各ステップの完了を明確に確認できる
3. **ロールバック手順**: 問題発生時に元の状態に戻す手順が含まれている
4. **並行開発対応**: 他のフェーズと同時進行できるよう、依存関係が明記されている
5. **見積もり時間付き**: 作業の規模感が事前にわかる
6. **コード付き**: コピー&ペーストで実装できる具体的なコードが含まれている

## 4. CHANGELOG の書き方

本プロジェクトでは [Keep a Changelog](https://keepachangelog.com/en/1.0.0/) 形式を採用する。

### カテゴリ定義

| カテゴリ | 用途 | 例 |
|---------|------|-----|
| **Added** | 新機能の追加 | 新しい API エンドポイント、新しいページ |
| **Changed** | 既存機能の変更 | API レスポンス形式の変更、UI の改善 |
| **Deprecated** | 将来削除予定の機能 | 旧 API の非推奨化 |
| **Removed** | 削除した機能 | レガシーコードの削除 |
| **Fixed** | バグ修正 | 認証フローのバグ修正 |
| **Security** | セキュリティに関する変更 | 脆弱性の修正、CSP の強化 |

### 書き方ルール

1. **最新のエントリを上に**: 新しい変更ほど上に記載する
2. **バージョン番号付き**: `## [X.Y.Z] - YYYY-MM-DD - Phase N: タイトル`
3. **Unreleased セクション**: リリース前の変更は `[Unreleased]` に記載する
4. **テーブル形式**: ファイル名と内容をテーブルで管理する
5. **概要セクション**: 各バージョンの冒頭に 1-2 文の概要を置く

### エントリ例

```markdown
## [46.0.0] - 2026-03-05 - Phase 44-46: Documentation

### 概要

ドキュメント整備。ドキュメントガイド、Next.js 16 移行ガイド、運用ドキュメント。

### Added

| ファイル | 内容 |
|---------|------|
| `docs/guides/DOCUMENTATION-GUIDE.md` | ドキュメント運用ガイド（Phase 44） |
| `docs/guides/NEXTJS16-MIGRATION.md` | Next.js 16 移行ガイド（Phase 45） |
| `docs/guides/OPERATIONS-GUIDE.md` | 運用ドキュメントガイド（Phase 46） |
```

## 5. SemVer（セマンティックバージョニング）

本プロジェクトでは [Semantic Versioning 2.0.0](https://semver.org/spec/v2.0.0.html) に準拠する。

### バージョン形式: MAJOR.MINOR.PATCH

| 要素 | 更新条件 | 例 |
|------|---------|-----|
| **MAJOR** | 後方互換性のない変更 | API のレスポンス形式変更、DB スキーマの破壊的変更 |
| **MINOR** | 後方互換性のある機能追加 | 新しいページ・API の追加、新しい設定項目 |
| **PATCH** | 後方互換性のあるバグ修正 | 表示バグの修正、typo の修正 |

### FDC プロジェクトでの運用

本プロジェクトではフェーズ完了時に MAJOR バージョンを上げる慣習を採用している:

- Phase 完了 = MAJOR バージョンアップ（例: v43.0.0 -> v46.0.0）
- フェーズ内の中間リリース = MINOR（例: v43.1.0）
- バグ修正 = PATCH（例: v43.0.1）

## 6. 500行ファイルルールとバレルファイルパターン

### 500行ルール

1つのファイルは原則 **500行以内** に収める。
500行を超えた場合は、以下の方法で分割を検討する:

- **コンポーネントの分割**: 大きなコンポーネントを子コンポーネントに分割する
- **ロジックの分離**: カスタムフック（`useXxx.ts`）やユーティリティ（`utils/xxx.ts`）に切り出す
- **型定義の分離**: 型定義が大きい場合は `types/xxx.ts` に分離する

### バレルファイルパターン

バレルファイル（`index.ts`）は、ディレクトリ内のモジュールを再エクスポートするファイルである。

```typescript
// lib/types/index.ts -- バレルファイル
export type { User, AppData } from './user';
export type { Task, TaskStatus, Suit } from './task';
export type { Workspace, WorkspaceMember } from './workspace';
```

#### バレルファイルの利点

- インポートパスが短くなる: `import { Task } from '@/lib/types'`
- モジュールの公開 API を制御できる
- リファクタリング時の影響範囲を限定できる

#### Turbopack での注意点

Next.js 16 の Turbopack では、同名ファイルの優先順位に注意が必要:

- `index.ts` と `index.tsx` が共存すると `.ts` が優先される
- `Component.tsx` と `Component/index.tsx` が共存するとエラーになる可能性がある

詳細は `docs/guides/NEXTJS16-MIGRATION.md` を参照のこと。

## 7. FDC-CORE.md 更新チェックリスト

フェーズ完了時、FDC-CORE.md を以下の順序で更新する:

### 更新項目

- [ ] **バージョン番号**: 冒頭の `v43.0.0` 形式のバージョンを更新
- [ ] **開発状況セクション**: `バージョン` の値を新バージョンに更新
- [ ] **フェーズ完了テーブル**: 完了したフェーズの行を追加（状態: 完了）
- [ ] **現在のPhase**: `現在のPhase` 行を最新フェーズに更新
- [ ] **次フェーズ**: `次フェーズ` 行を次のフェーズ番号に更新
- [ ] **技術的負債**: 新たに発見した技術的負債があれば追記
- [ ] **機能リスト**: 新機能を追加した場合、開発状況の機能リストに追記
- [ ] **末尾のメタ情報**: `Last Updated`, `Version`, `Status` を更新

### 更新例

```diff
- **バージョン**: v43.0.0
+ **バージョン**: v46.0.0

- **現在のPhase**: Phase 43 完了
+ **現在のPhase**: Phase 46 完了

- **次フェーズ**: Phase 44
+ **次フェーズ**: Phase 47

- **Version**: v43.0.0
+ **Version**: v46.0.0

- **Status**: Phase 43 完了
+ **Status**: Phase 46 完了
```
