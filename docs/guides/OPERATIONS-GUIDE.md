# Operations Guide

> FDC Modular Starter の運用・インシデント対応・SLO/SLA ガイド

---

## 1. 運用ドキュメントの種類

FDC プロジェクトの運用ドキュメントは以下の 3 種類で構成される。

| 種類 | 目的 | 例 |
|------|------|-----|
| **ランブック（Runbook）** | 定型作業の手順書。再現可能なステップバイステップ形式 | デプロイ手順、DB マイグレーション手順 |
| **運用ドキュメント（Ops Doc）** | 運用方針・体制・ポリシーの定義 | 本ドキュメント、インシデント対応ガイド |
| **トラブルシューティング** | 問題発生時の調査・解決手順 | デバッグガイド、認証トラブルシューティング |

### 使い分け

- **定型作業** -> ランブック（`docs/runbooks/`）
- **方針・体制の定義** -> 運用ドキュメント（`docs/guides/`）
- **問題の調査** -> トラブルシューティング（`docs/guides/DEBUG-GUIDE.md` 等）

## 2. インシデント対応フロー

インシデント（障害）発生時は以下の 5 ステップで対応する。

### 2.1 対応フロー概要

```
[1. 検知] -> [2. 初動対応] -> [3. 調査] -> [4. 復旧] -> [5. ポストモーテム]
```

### Step 1: 検知（Detect）

インシデントの発見経路:

| 経路 | 説明 |
|------|------|
| アラート通知 | ヘルスチェック失敗、エラーレート閾値超過 |
| ユーザー報告 | サポートチャネル経由の障害報告 |
| 監視ダッシュボード | Vercel Analytics / Supabase Dashboard の異常値 |
| ログ監視 | サーバーログの異常パターン検出 |

検知したら直ちに Step 2 へ進む。

### Step 2: 初動対応（Initial Response）

1. **重要度を判定する**

| レベル | 説明 | 目標復旧時間 |
|--------|------|-------------|
| P1（Critical） | 全面停止。ユーザーが一切利用できない | 1 時間 |
| P2（Major） | 主要機能の障害。一部機能が利用不可 | 4 時間 |
| P3（Minor） | 軽微な問題。回避策あり | 24 時間 |
| P4（Low） | 改善要望レベル | 次回リリース |

2. **対応チームに通知する**（P1/P2 の場合は即座に）
3. **インシデントチャネルを開設する**（Slack 等）
4. **ステータスページを更新する**（該当する場合）

### Step 3: 調査（Investigate）

1. **影響範囲を特定する**: どの機能・ユーザーが影響を受けているか
2. **タイムラインを作成する**: いつから問題が発生しているか
3. **ログを確認する**: Vercel Functions ログ、Supabase ログ、ブラウザコンソール
4. **直近の変更を確認する**: 最後のデプロイ内容、DB マイグレーション

### Step 4: 復旧（Recover）

復旧の優先順位:

1. **ロールバック**: 直前のデプロイに戻す（Vercel: Promote previous deployment）
2. **ホットフィックス**: 原因が明確な場合、最小限の修正をデプロイ
3. **回避策の適用**: 根本修正が時間を要する場合、一時的な回避策を適用

```
Vercel ロールバック手順:
1. Vercel Dashboard > Deployments
2. 正常だった前回のデプロイを選択
3. "Promote to Production" をクリック
```

### Step 5: ポストモーテム（Postmortem）

P1/P2 インシデントの後は必ずポストモーテムを作成する。

## 3. ポストモーテムテンプレート

```markdown
# Postmortem: [インシデントタイトル]

## サマリー

| 項目 | 内容 |
|------|------|
| 発生日時 | YYYY-MM-DD HH:MM (JST) |
| 復旧日時 | YYYY-MM-DD HH:MM (JST) |
| 影響範囲 | 影響を受けたユーザー数・機能 |
| 重要度 | P1 / P2 / P3 / P4 |
| 担当者 | 対応者名 |

## タイムライン

| 時刻 (JST) | イベント |
|-------------|---------|
| HH:MM | 問題が最初に報告される |
| HH:MM | 調査開始 |
| HH:MM | 原因を特定 |
| HH:MM | 修正をデプロイ |
| HH:MM | 復旧を確認 |

## 根本原因（Root Cause）

問題の根本原因を技術的に説明する。
- 何が壊れたか
- なぜ壊れたか
- なぜ事前に検知できなかったか

## 再発防止策（Prevention Measures）

| 対策 | 担当 | 期限 | 状態 |
|------|------|------|------|
| [対策内容 1] | [担当者] | [期限] | 未着手 / 完了 |
| [対策内容 2] | [担当者] | [期限] | 未着手 / 完了 |

## 学んだこと（Lessons Learned）

- 良かった点（対応で効果的だったこと）
- 改善点（次回はこうすべきだったこと）
```

### ポストモーテムの原則

- **非難しない（Blameless）**: 個人を責めるのではなく、システム・プロセスの改善に焦点を当てる
- **事実ベース**: 推測ではなく、ログやデータに基づいた記述を行う
- **アクション駆動**: 再発防止策には必ず担当者と期限を設定する

## 4. アラート設計の原則

効果的なアラートを設計するための原則。

### アラートの分類

| レベル | アクション | 例 |
|--------|-----------|-----|
| **Critical** | 即座に対応が必要 | サービス全面停止、データ損失の危険 |
| **Warning** | 数時間以内に確認が必要 | エラーレートの上昇、ディスク使用率 80% |
| **Info** | 次回の業務時間内に確認 | デプロイ成功通知、定期レポート |

### 良いアラートの条件

1. **アクション可能**: アラートを受け取ったら、具体的に何をすべきかが明確
2. **誤報が少ない**: ノイズが多いアラートは無視されるようになる
3. **十分な情報**: 調査の出発点となる情報（URL、エラーメッセージ、時刻）を含む
4. **適切な粒度**: 1 つの障害で大量のアラートが飛ばない（集約・重複排除）
5. **閾値の根拠**: なぜその閾値なのかが説明できる

### FDC でのアラート設定例

| 対象 | 条件 | レベル |
|------|------|--------|
| ヘルスチェック API | 2 回連続失敗 | Critical |
| API エラーレート | 5 分間で 10% 超過 | Warning |
| ビルド | CI ビルド失敗 | Warning |
| 依存関係 | セキュリティ脆弱性検出 | Info |

## 5. オンコール体制

### ローテーション

- オンコール担当者は 1 週間単位でローテーションする
- 交代は毎週月曜日 09:00 (JST)
- 引き継ぎ時に前週のインシデント・注意事項を共有する

### エスカレーション

```
Level 1: オンコール担当者（15 分以内に応答）
  |
  +-- 30 分以内に解決しない場合
  |
Level 2: テックリード（追加支援）
  |
  +-- 1 時間以内に解決しない場合
  |
Level 3: プロジェクトマネージャー（ビジネス判断）
```

### オンコール担当者の責務

1. **P1/P2 アラートに 15 分以内に応答する**
2. **インシデント対応フローに従って初動対応する**
3. **必要に応じてエスカレーションする**
4. **インシデント発生時のタイムラインを記録する**
5. **翌営業日に引き継ぎメモを更新する**

## 6. ステータスページ

ユーザーにサービスの稼働状況を伝えるためのステータスページの考え方。

### ステータスレベル

| ステータス | 表示 | 説明 |
|-----------|------|------|
| Operational | 正常稼働 | すべてのサービスが正常 |
| Degraded Performance | パフォーマンス低下 | 動作するが遅延がある |
| Partial Outage | 一部障害 | 一部の機能が利用不可 |
| Major Outage | 大規模障害 | サービス全体が利用不可 |
| Under Maintenance | メンテナンス中 | 計画的な停止 |

### 更新のタイミング

- インシデント検知直後: ステータスを変更し、調査中であることを告知
- 30 分ごと: 進捗を更新（復旧見込み時間があれば記載）
- 復旧後: ステータスを正常に戻し、事後レポートへのリンクを追加

## 7. SLI / SLO / SLA

サービスの信頼性を定量的に管理するための 3 つの指標。

### 定義と違い

| 用語 | 正式名称 | 定義 | 例 |
|------|---------|------|-----|
| **SLI** | Service Level Indicator | サービスの品質を測定する指標 | リクエスト成功率、レスポンスタイム |
| **SLO** | Service Level Objective | SLI に対する目標値（内部目標） | 成功率 99.9%、P95 レスポンス < 500ms |
| **SLA** | Service Level Agreement | 顧客との契約上の保証値 | 月間稼働率 99.5%（違反時に返金） |

### 関係性

```
SLA <= SLO（SLO は SLA より厳しく設定する）

例:
  SLA: 月間稼働率 99.5%（顧客への約束）
  SLO: 月間稼働率 99.9%（内部目標）
  SLI: 実際の稼働率（測定値）
```

SLO を SLA より厳しく設定することで、SLA 違反の前に問題を検知できるバッファを確保する。

### FDC プロジェクトでの SLI 例

| SLI | 測定方法 | SLO 目標 |
|-----|---------|---------|
| 可用性（Availability） | ヘルスチェック API の成功率 | 99.9% / 月 |
| レスポンスタイム | API の P95 レスポンスタイム | 500ms 以下 |
| エラーレート | 5xx レスポンスの割合 | 0.1% 以下 |
| ビルド成功率 | CI パイプラインの成功率 | 95% 以上 |

### SLO の設計方針

1. **ユーザー体験に基づく**: 技術指標ではなく、ユーザーが体感する品質を測定する
2. **測定可能**: 自動的に計測できる指標を選ぶ
3. **現実的な目標**: 100% を目標にしない（コストが指数関数的に増大する）
4. **定期的な見直し**: 四半期ごとに SLO を見直し、実態に合わせて調整する

## 8. エラーバジェット

エラーバジェットは SLO の裏返しであり、許容されるダウンタイムの量を表す。

### エラーバジェットの計算

```
エラーバジェット = 1 - SLO 目標値

例: SLO = 99.9% の場合
  エラーバジェット = 0.1%
  月間（30 日）の許容ダウンタイム = 30 * 24 * 60 * 0.001 = 43.2 分
```

| SLO 目標 | エラーバジェット | 月間許容ダウンタイム |
|---------|----------------|-------------------|
| 99.0% | 1.0% | 7 時間 12 分 |
| 99.5% | 0.5% | 3 時間 36 分 |
| 99.9% | 0.1% | 43 分 |
| 99.95% | 0.05% | 22 分 |
| 99.99% | 0.01% | 4 分 |

### エラーバジェットの活用

- **バジェット残量が多い**: 新機能のリリースやリスクのある変更を積極的に実施できる
- **バジェット残量が少ない**: 安定性を優先し、リスクの高い変更を延期する
- **バジェットを使い切った**: 新機能の開発を凍結し、信頼性向上に集中する

### 判断フレームワーク

```
エラーバジェット残量 > 50%  -> 通常の開発速度で進行
エラーバジェット残量 > 20%  -> リスクの高い変更にはレビューを追加
エラーバジェット残量 < 20%  -> 安定性改善を最優先にする
エラーバジェット残量 = 0%   -> 機能開発を凍結、信頼性エンジニアリングに集中
```
