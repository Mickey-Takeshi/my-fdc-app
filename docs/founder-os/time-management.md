# ファウンダー時間管理設計（Phase 76）

## 0. 概要

ファウンダーの時間は最も希少なリソースである。
委任、集中時間、情報フィルタリング、バーンアウト防止の4つの柱で
ファウンダーの生産性を最大化する設計を定義する。

---

## 1. 委任（Delegation）

### 1.1 委任マトリクス

| タスク | 委任先 | 委任レベル | 備考 |
|--------|--------|-----------|------|
| 日常カスタマーサポート | CS担当 | L5（完全自律） | 対応ガイドライン整備済み |
| コードレビュー | テックリード | L4（報告不要） | マージ権限委譲 |
| 一次面接 | チームリーダー | L3（結果報告） | 評価基準シート使用 |
| バグトリアージ | テックリード | L4（報告不要） | 重大バグのみエスカレーション |
| SNS/ブログ運営 | マーケ担当 | L3（結果報告） | ブランドガイドライン準拠 |
| 経理・請求処理 | 経理担当/外注 | L4（報告不要） | 月次レポートで確認 |
| 資金調達 | **委任しない** | - | ファウンダー固有 |
| 事業戦略 | **委任しない** | - | ファウンダー固有 |
| 主要顧客との関係構築 | **委任しない** | - | ファウンダー固有 |
| 採用最終判断 | **委任しない** | - | ファウンダー固有 |
| ビジョン策定 | **委任しない** | - | ファウンダー固有 |

### 1.2 5段階委任レベル

| レベル | 名称 | 説明 | ファウンダーの関与 |
|--------|------|------|-------------------|
| L1 | リサーチ | 情報を調べてファウンダーに報告 | 判断・実行はファウンダー |
| L2 | 提案 | 選択肢を整理して提案 | 最終判断はファウンダー |
| L3 | 実行+報告 | 実行して結果を報告 | 事後確認のみ |
| L4 | 完全実行 | 実行し報告は定期のみ | 月次/週次レポートで確認 |
| L5 | 完全自律 | すべて自律的に実行 | 異常時のみ関与 |

### 1.3 委任プロセス

```
委任の4ステップ
==============

Step 1: 説明（Explain）
  - タスクの目的と期待する成果を明確に伝える
  - 判断基準と制約条件を共有する
  - 質問を受け付け、理解を確認する
  期間: 1-2回

Step 2: 同行（Shadow）
  - ファウンダーが実行する様子を見せる
  - 判断のポイントをリアルタイムで解説する
  - 担当者が質問できる環境を作る
  期間: 1-2週間

Step 3: 監視（Monitor）
  - 担当者が実行し、ファウンダーがレビューする
  - フィードバックを提供し改善を促す
  - 徐々にレビュー頻度を下げる
  期間: 2-4週間

Step 4: 完全委任（Full Delegate）
  - 担当者が自律的に実行する
  - 定期報告（週次/月次）のみ確認
  - 異常時のエスカレーションルールを設定
  期間: 継続
```

### 1.4 絶対に委任しないもの

```
ファウンダーが直接行うべき事項:

1. ビジョンと戦略
   - 会社のビジョン・ミッション策定
   - 中長期事業戦略の決定
   - ピボット判断

2. 資金調達
   - 投資家とのリレーション構築
   - ピッチ・プレゼンテーション
   - 条件交渉

3. 主要な人事決定
   - 幹部採用の最終面接
   - 解雇の最終判断
   - 組織文化の方向性

4. 主要顧客との関係
   - トップ10顧客との定期的な対話
   - 重大クレームの最終対応
   - 戦略的パートナーシップ交渉

5. プロダクトビジョン
   - プロダクトの長期ロードマップ
   - 重要な機能の方向性決定
   - ユーザー体験の最終判断
```

### 2.2 集中モード設定

| 設定項目 | 設定内容 |
|---------|---------|
| Slack | Do Not Disturb（DND）モード ON |
| メール | 自動返信「集中作業中です。緊急の場合は電話をお願いします」 |
| カレンダー | 予定をブロック（「Focus Time - Do Not Book」） |
| 通知 | クリティカルのみ許可（本番障害アラートのみ） |
| 物理環境 | ヘッドフォン着用（チームへの「集中中」のシグナル） |

### 2.3 集中時間の使い方ガイド

```
集中時間ブロックの活用法:

朝の集中ブロック（07:00-09:00）: 戦略的思考
  - 週次/月次の計画
  - プロダクト戦略の検討
  - 重要なドキュメント作成
  - 投資家向け資料の準備

午前の集中ブロック（09:15-12:00）: ディープワーク
  - コーディング/プロトタイピング
  - 顧客インタビューの分析
  - 技術的な問題解決
  - 新機能の設計

午後の集中ブロック（16:00-18:00）: 整理と準備
  - 翌日のタスク整理
  - レビュー待ちの処理
  - ドキュメントの更新
  - 非同期コミュニケーションの対応

注意事項:
  - 集中ブロック中は25分作業+5分休憩（ポモドーロ）
  - 最低でも90分の連続時間を確保する
  - 集中ブロックのスキップは週2回まで許容
  - 水曜日のNo-Meeting Dayは最優先で死守する
```

## 4. バーンアウト防止

### 4.1 週次セルフチェック

毎週金曜日の振り返り時に確認する:

| チェック項目 | 基準 | 今週 |
|-------------|------|------|
| 睡眠 | 7時間以上を週5日確保 | [ ] Yes / [ ] No |
| 運動 | 週3回以上の運動 | [ ] Yes / [ ] No |
| 仕事以外の活動 | 趣味・家族・友人との時間確保 | [ ] Yes / [ ] No |
| 集中時間 | 計画した集中ブロックの80%以上を確保 | [ ] Yes / [ ] No |
| 週末の休息 | 週末に少なくとも1日は完全オフ | [ ] Yes / [ ] No |

**判定基準**:
- 5/5 達成: 良好 - 現状維持
- 3-4/5 達成: 注意 - 来週の改善計画を立てる
- 2/5 以下: 警告 - 即座にスケジュールを見直す

### 4.2 警告サインと対応

| 警告サイン | レベル | 対応アクション |
|-----------|--------|--------------|
| 睡眠不足が3日連続 | 注意 | 翌日の朝の集中ブロックを睡眠に充てる |
| 週末に2週連続で作業 | 注意 | 来週末は完全オフを死守、タスクの委任を増やす |
| 集中時間が2週連続で50%未満 | 警告 | 会議の棚卸し、不要な会議を断る |
| モチベーション低下が1週間以上 | 警告 | メンターや同業ファウンダーに相談、環境変更 |
| 身体的不調（頭痛、肩こり、胃の不調） | 警告 | 医療機関受診、1-2日の完全休暇取得 |
| 意思決定の質の低下を自覚 | 緊急 | 3日以上の休暇を即座に取得、バックアップに引き継ぎ |
| バーンアウト症状（無力感、虚無感） | 緊急 | 1週間以上の休暇、専門家への相談、復帰計画の策定 |

### 4.3 休暇ポリシー

| 項目 | ルール |
|------|--------|
| 最低取得日数 | 四半期ごとに最低5営業日 |
| 事前通知 | 2週間前までにチームに通知 |
| バックアップ | 不在時の意思決定者を必ず指名 |
| 接続 | 休暇中は完全オフライン（緊急連絡は電話のみ） |
| 引き継ぎ | 休暇前に進行中タスクの引き継ぎドキュメントを作成 |
