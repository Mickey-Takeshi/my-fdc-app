# プロセス設計（Phase 75）

## 0. 概要

ファウンダー主導のスタートアップにおける定例会議、意思決定フレームワーク、
OKR運用、ドキュメント文化の設計書。少人数チームでの効率的なプロセスを定義する。

## 2. 意思決定フレームワーク

### 2.1 RACI マトリクス

FDC における主要な意思決定の責任分担:

| 意思決定事項 | ファウンダー | テックリード | チームリーダー | メンバー |
|-------------|:----------:|:----------:|:------------:|:-------:|
| 採用（Hiring） | A | C | R | I |
| 技術選定（Tech Selection） | A | R | C | I |
| 機能優先度（Feature Priority） | R/A | C | C | I |
| リリース判断（Release） | I | R/A | C | I |
| 予算承認（Budget） | R/A | C | I | I |
| アーキテクチャ変更 | A | R | C | I |
| 顧客対応方針 | R/A | I | C | I |
| セキュリティ対応 | A | R | C | I |

```
R = Responsible（実行責任者）
A = Accountable（最終承認者）
C = Consulted（相談先）
I = Informed（報告先）
```

### 2.3 優先度基準

| 優先度 | 定義 | 対応時間 | 例 |
|--------|------|---------|-----|
| 即時（Immediate） | サービス停止・データ損失のリスク | 1時間以内 | 本番障害、セキュリティインシデント |
| 高（High） | 収益への直接影響、顧客体験の大幅な低下 | 当日中 | 決済エラー、主要機能のバグ |
| 中（Medium） | 改善の余地があるが緊急ではない | 今週中 | 機能改善、パフォーマンス最適化 |
| 低（Low） | 将来的に対応すべき事項 | 次スプリント以降 | 技術負債、UI微調整 |

### 3.2 OKR 階層

```
会社OKR（ファウンダー設定）
│
├── Objective: PMFを達成し持続的な成長基盤を構築する
│   ├── KR1: WAT を 150 に到達させる（現在: 100）
│   ├── KR2: MRR 300万円を達成する（現在: 200万円）
│   └── KR3: NPS 50以上を維持する（現在: 45）
│
└── チームOKR（チームリーダー設定）
    │
    ├── プロダクトチーム
    │   ├── Objective: ユーザーが毎日使いたくなるプロダクトにする
    │   │   ├── KR1: DAU/MAU比率を 40% に向上（現在: 30%）
    │   │   ├── KR2: オンボーディング完了率を 80% に向上（現在: 60%）
    │   │   └── KR3: 機能利用率 Top3 の平均を 70% に向上
    │   │
    │   └── 個人OKR（メンバー設定）
    │       └── Objective: オンボーディング体験を最適化する
    │           ├── KR1: ウィザード完了率を 85% に向上
    │           ├── KR2: Aha Moment到達時間を 5分以内に短縮
    │           └── KR3: オンボーディング後7日リテンションを 70% に
    │
    └── エンジニアリングチーム
        └── Objective: 高速で安定したデリバリーを実現する
            ├── KR1: デプロイ頻度を週5回に向上（現在: 週3回）
            ├── KR2: 本番障害を月1件以下に削減（現在: 月3件）
            └── KR3: テストカバレッジを 80% に向上（現在: 65%）
```

## 4. ドキュメント文化

### 4.1 原則

| 原則 | 説明 |
|------|------|
| Write First | 議論の前にドキュメントを書く。会議はドキュメントのレビューから始める |
| Single Source of Truth | 各情報には唯一の正となるドキュメントが存在する |
| バージョン管理 | すべてのドキュメントは Git で管理し、変更履歴を追跡可能にする |
| 四半期レビュー | 四半期ごとに全ドキュメントの棚卸しを実施し、陳腐化を防止する |

### 4.2 ドキュメント種別と管理

| 種別 | 保管場所 | 更新頻度 | 責任者 |
|------|---------|---------|--------|
| 技術仕様 | Git (docs/) | 変更時 | テックリード |
| 議事録 | Notion / Git | 会議後24時間以内 | ファシリテーター |
| ADR（Architecture Decision Record） | Git (docs/adr/) | 決定時 | 提案者 |
| ランブック | Git (docs/runbooks/) | 変更時 | 実行者 |
| OKR | 共有スプレッドシート | 週次 | 各OKR所有者 |
| KPIレポート | ダッシュボード | 自動更新 | データ担当 |

### 4.3 議事録ルール

```
議事録の基本ルール:

1. 形式
   - 日時、参加者、議題を明記
   - 決定事項とアクションアイテムを明確に分離
   - アクションアイテムには担当者と期限を記載

2. タイミング
   - 会議中にリアルタイムで記録
   - 会議後24時間以内に共有

3. 保管
   - Notion または Git のどちらかに統一
   - 検索可能な形式で管理
   - タグ/カテゴリで分類

4. テンプレート
   ──────────────────────
   会議名: [名称]
   日時: YYYY-MM-DD HH:MM
   参加者: [名前, 名前, ...]
   ファシリテーター: [名前]
   記録者: [名前]

   議題:
   1. [議題1]
   2. [議題2]

   議論内容:
   - [要点を箇条書き]

   決定事項:
   - [決定1]
   - [決定2]

   アクションアイテム:
   - [ ] [内容] / [担当] / [期限]
   - [ ] [内容] / [担当] / [期限]

   次回予定:
   - 日時: YYYY-MM-DD HH:MM
   - 議題: [予定]
   ──────────────────────
```
